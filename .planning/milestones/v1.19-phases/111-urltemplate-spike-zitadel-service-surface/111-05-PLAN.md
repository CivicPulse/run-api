<objective>
Produce `111-SCOPE-AUDIT.md` — a grep-based inventory of every ZITADEL API endpoint the runtime `ZitadelService` calls today, mapped to the minimum ZITADEL role required per the v2 API docs. If the audit shows the runtime service-account is scoped more broadly than `ORG_USER_MANAGER` + `ORG_PROJECT_USER_GRANT_EDITOR` on the CivicPulse org (e.g. `IAM_OWNER`), narrow it via the ZITADEL Console, make any additive changes to `scripts/bootstrap-zitadel.py` so future fresh bootstraps land on the narrowed shape, and ship an integration test proving `ensure_human_user`, `create_invite_code`, and `ensure_project_grant` all still work under the narrowed scope.

Per D-SCOPE-01, this phase targets ONLY the runtime service-account (the OIDC client used by `ZitadelService._get_token`), NOT the bootstrap PAT (which is a dev-time tool; re-scoping it is deferred to a future phase).
</objective>

<task id="111-05-T3" type="execute">
<action>
Verify the current scope of the runtime service-account against the dev ZITADEL. Run:

    # Get a fresh runtime service-account token (reads the same envs Plan 01 uses)
    TOKEN=$(curl -s -X POST "${ZITADEL_URL}/oauth/v2/token" \
      -d "grant_type=client_credentials" \
      -d "client_id=${ZITADEL_SERVICE_CLIENT_ID}" \
      -d "client_secret=${ZITADEL_SERVICE_CLIENT_SECRET}" \
      -d "scope=openid urn:zitadel:iam:org:project:id:zitadel:aud" | jq -r .access_token)

    # Fetch the memberships (role assignments) of the calling identity
    curl -s -H "Authorization: Bearer ${TOKEN}" \
      "${ZITADEL_URL}/auth/v1/users/me/memberships/_search" \
      -X POST -d '{}' -H 'Content-Type: application/json' | jq .
