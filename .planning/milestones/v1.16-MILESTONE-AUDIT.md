# Milestone v1.16 Audit — Email Delivery Foundation

**Audited:** 2026-04-08  
**Status:** TECH DEBT REVIEW

## Summary

All six v1.16 phases are complete and phase-level verification reports passed,
and the invite-delivery chain is wired end to end across provider setup, async
send, webhook reconciliation, operations docs, and the admin pending-invites
UI.

The previously identified milestone blockers are now closed:

- The pending-invite revoke action no longer parses JSON from a `204 No Content`
  response, so staff can complete remediation from the UI.
- Mailgun webhook reconciliation now preserves delivery truth for org-less
  invite attempts instead of dropping those events.

The remaining concerns are non-blocking operational debt: missing Nyquist
validation artifacts and the absence of live-provider smoke tests in this
workspace.

## Cross-Phase Integration (12/12 Wired)

Confirmed working seams:

1. `app/core/config.py` -> `app/services/email_provider.py` -> `app/services/invite_email.py` wires env-scoped Mailgun selection, single-recipient payload construction, app-owned invite rendering, and public accept URL generation.
2. `app/services/invite.py` -> `app/tasks/invite_tasks.py` commits invite state before queueing and preserves idempotent skip/failure behavior.
3. `app/api/v1/volunteers.py` reuses `InviteService.create_invite`, so volunteer invite mode shares the same transactional path.
4. `app/tasks/invite_tasks.py` -> `app/services/email_delivery.py` -> `app/models/email_delivery_attempt.py` persists canonical attempt rows and projects latest invite delivery state.
5. `app/api/v1/mailgun_webhooks.py` -> `app/services/mailgun_webhook.py` verifies Mailgun signatures before mutating state and correlates by provider message id rather than recipient email.
6. `app/api/v1/invites.py` -> `web/src/hooks/useInvites.ts` -> `web/src/routes/campaigns/$campaignId/settings/members.tsx` now agrees on the bare `Invite[]` list contract and delivery fields.
7. `app/api/v1/invites.py` public invite metadata remains wired to `web/src/routes/invites/$token.tsx` and the existing auth redirect/acceptance flow.
8. `docs/email-operations-runbook.md` and `docs/zitadel-email-runbook.md` keep the CivicPulse-vs-ZITADEL support boundary consistent.

Closed integration gaps:

1. `web/src/hooks/useInvites.ts` now treats revoke as a `204 No Content` mutation and the members UI remains wired to the same invite-query invalidation path.
2. `app/api/v1/mailgun_webhooks.py` now reconciles org-less invite attempts instead of returning early before projection.
