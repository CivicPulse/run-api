3. For each route, create a test using `test.describe` and `test()`:
   ```typescript
   for (const route of ROUTES) {
     test(`a11y scan: ${route.name}`, async ({ page, makeAxeBuilder }, testInfo) => {
       await page.goto(route.path);
       // Wait for any loading states to resolve
       await page.waitForLoadState('networkidle');

       const results = await makeAxeBuilder().analyze();

       // Save full results as JSON artifact (per D-04)
       await testInfo.attach(`axe-${route.name}`, {
         body: JSON.stringify(results, null, 2),
         contentType: 'application/json',
       });

       // Gate on zero critical/serious violations (per D-02)
       const critical = results.violations.filter(
         (v) => v.impact === 'critical' || v.impact === 'serious'
       );

       // Log moderate/minor as warnings
       const warnings = results.violations.filter(
         (v) => v.impact === 'moderate' || v.impact === 'minor'
       );
       if (warnings.length > 0) {
         console.warn(
           `[${route.name}] ${warnings.length} moderate/minor violations:`,
           warnings.map((w) => `${w.id}: ${w.description}`).join(', ')
         );
       }

       expect(critical, `Critical/serious violations on ${route.name}: ${critical.map(v => `${v.id}: ${v.help}`).join('; ')}`).toEqual([]);
     });
   }
   ```

**Part A: Fix axe-core violations (common violations to expect and fix):**

1. **Missing button labels (button-name):**

   - Check if any `<Link>` components render icon-only without text content
   - Add `aria-label` to icon-only links

3. **Color contrast (color-contrast):**
   - Should be mostly fixed by Plan 01's muted-foreground adjustment
   - If additional violations found, fix specific CSS classes or component styles

4. **Missing form labels (label):**
   - Check DataTable filter inputs for `<label>` or `aria-label`
   - Check search inputs across the app

5. **Image alt text (image-alt):**
   - Check any `<img>` tags for missing `alt` attributes
   - For decorative images, add `alt=""` and `aria-hidden="true"`

6. **Heading hierarchy (heading-order):**
   - Per D-09: audit page headings. Each route should have one h1, sections use h2
   - Fix routes that skip heading levels (e.g., h1 to h3)

7. **TooltipIcon accessibility:**
   - Add `aria-describedby` linkage between the trigger and tooltip content
   - Ensure the Popover trigger has an accessible name

**Part B: Implement D-10 focus management patterns (per D-10):**

D-10 requires explicit focus management for three cases. Use the `FocusScope` component and `useFocusScope` hook created in Plan 01:

8. **Form submission focus management (per D-10):**
   - Identify form components that show success/error messages after submission (e.g., campaign settings forms, voter import, turf form)
   - After successful form submission: programmatically focus the success toast/message element so screen readers announce it. Use `useFocusScope` hook's `focusElement` to target the success message, or ensure the success message container has `role="status"` and `aria-live="polite"` so it is announced without explicit focus move
   - After failed form submission with validation errors: focus the first error message or the first invalid field. Use `el.focus()` on the first field with `aria-invalid="true"`, or use `useFocusScope` hook's `focusElement('[aria-invalid="true"]')` to focus the first invalid input
   - At minimum, apply this pattern to: settings forms (general, members), turf form (TurfForm.tsx), and any form that displays inline error/success messages

9. **Delete action focus management (per D-10):**
   - Identify delete actions in the app (e.g., delete turf, remove member, delete campaign in danger zone)
   - After a delete confirmation dialog closes and the item is removed: focus the next item in the list, or if the list is now empty, focus the list heading or empty state message
   - Do NOT just let focus drop to `<body>` after deletion -- this disorients screen reader users
   - Implementation pattern: Before showing the delete dialog, store a ref to the next sibling item. After deletion completes and dialog closes, call `.focus()` on that element. If using DataTable rows, focus the row at the same index (or last row if deleted item was last)
   - At minimum, apply this pattern to: member removal (settings/members), turf deletion, and any list item with a delete action

10. **Dialog close focus restoration (per D-10):**
    - shadcn Dialog already handles this via Radix -- verify it works correctly by checking that focus returns to the trigger button when a dialog is closed via Escape or close button
    - No custom code needed if shadcn Dialog is used consistently -- just verify during the axe scan that no custom modals bypass this

This task is REACTIVE for Part A -- run the scan first, then fix what it finds. Part B is PROACTIVE -- implement D-10 patterns regardless of axe findings (axe cannot detect focus management issues).
