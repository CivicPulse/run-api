# Roadmap: v1.10 Import Recovery

## Overview

Make import processing crash-safe across worker restarts, deploy rollouts, and pod evictions. Orphaned imports self-heal without manual intervention. This is a focused backend patch: add staleness tracking, build a recovery engine with advisory lock protection, harden the completion path, and cover it all with unit and integration tests.

## Phases

- [ ] **Phase 56: Schema & Orphan Detection** - Migration for last_progress_at, configurable staleness threshold, detection query, and structured logging
- [ ] **Phase 57: Recovery Engine & Completion Hardening** - Worker startup recovery scan, advisory lock guard, crash-resume from last_committed_row, and hardened finalization
- [ ] **Phase 58: Test Coverage** - Unit tests for detection/requeue logic, integration test for crash-resume, and duplicate voter regression assertions

## Phase Details

### Phase 56: Schema & Orphan Detection
**Goal**: The system can identify imports stuck in PROCESSING with stale progress and log detection events
**Depends on**: Nothing (first phase of milestone)
**Requirements**: ORPH-01, ORPH-02, ORPH-03, OBSV-01
**Success Criteria** (what must be TRUE):
  1. import_jobs table has a last_progress_at column that is updated after each committed batch
  2. A detection query correctly identifies imports in PROCESSING where last_progress_at exceeds the configurable staleness threshold
  3. Staleness threshold is configurable via application settings with a 5-minute default
  4. Structured log entries are emitted when orphaned imports are detected

### Phase 57: Recovery Engine & Completion Hardening
**Goal**: Orphaned imports are automatically reclaimed, resumed from where they left off, and finalization failures produce clear error states
**Depends on**: Phase 56
**Requirements**: RECV-01, RECV-02, RECV-03, RECV-04, HARD-01, HARD-02
**Success Criteria** (what must be TRUE):
  1. Worker startup triggers a recovery scan that re-queues orphaned imports as fresh Procrastinate tasks
  2. Re-queued imports resume processing from last_committed_row without producing duplicate voters
  3. pg_try_advisory_lock prevents two workers from executing the same import concurrently
  4. Imports in COMPLETED, FAILED, or CANCELLED status are never reclaimed by the recovery scan
  5. A finalization failure transitions the import to a distinct error state instead of leaving it stuck in PROCESSING
**Plans**: TBD

### Phase 58: Test Coverage
**Goal**: Recovery logic is verified by automated tests covering detection, resume, and data integrity
**Depends on**: Phase 57
**Requirements**: TEST-01, TEST-02, TEST-03, TEST-04, TEST-05
**Success Criteria** (what must be TRUE):
  1. Unit tests verify that stale PROCESSING imports are detected and queued for recovery
  2. Unit tests verify that completed, cancelled, and failed imports are never reclaimed
  3. Unit tests verify that imports with fresh progress are not reclaimed
  4. An integration test simulates crash-resume: partial commit, worker death, recovery scan, and successful completion
  5. A regression assertion confirms no duplicate voters exist after resume and the final row count matches the source file
