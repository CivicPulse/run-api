# Milestone v1.10: Import Recovery

**Status:** ✅ SHIPPED 2026-04-01
**Phases:** 56-58
**Total Plans:** 3

## Overview

Make import processing crash-safe across worker restarts, deploy rollouts, and pod evictions. Orphaned imports now self-heal without manual intervention through application-owned progress tracking, startup recovery scans, advisory-lock protection, and targeted automated coverage.

## Phases

### Phase 56: Schema & Orphan Detection

**Goal**: Identify imports stuck in `PROCESSING` using application-owned progress timestamps and emit recovery-ready diagnostics.
**Depends on**: Nothing (first phase of v1.10)
**Plans**: 1 plan

Plans:

- [x] 56-01: Add import recovery metadata, configurable orphan threshold, and persisted orphan diagnostics

**Details:**
- `import_jobs` now stores `last_progress_at`, `source_exhausted_at`, `orphaned_at`, and `orphaned_reason`
- `import_orphan_threshold_minutes` is configurable in app settings with a safe default
- Durable progress timestamps are updated only after committed import work
- Orphan scans persist recovery-ready metadata instead of depending on queue internals

### Phase 57: Recovery Engine & Completion Hardening

**Goal**: Reclaim orphaned imports safely, resume from `last_committed_row`, and harden finalization so completed work cannot stay stuck.
**Depends on**: Phase 56
**Plans**: 1 plan

Plans:

- [x] 57-01: Add worker-startup recovery orchestration, advisory locking, and exhausted-import finalization

**Details:**
- Worker startup scans for stale imports and enqueues fresh `recover_import` tasks
- Recovery resumes from `last_committed_row` and skips terminal or fresh jobs
- `pg_try_advisory_lock` protects both normal processing and recovery/finalization
- Exhausted imports can finalize directly during recovery instead of remaining stuck in `PROCESSING`

### Phase 58: Test Coverage

**Goal**: Verify stale-import detection, crash-resume recovery, and duplicate-prevention with automated tests.
**Depends on**: Phase 57
**Plans**: 1 plan

Plans:

- [x] 58-01: Add unit and integration coverage for detection, recovery, and duplicate-prevention

**Details:**
- Unit coverage verifies stale detection, worker-startup queueing, and skip logic for terminal or fresh jobs
- Integration-marked recovery flow verifies resume begins after `last_committed_row`
- Regression assertions confirm final row counts stay correct and already committed rows are not replayed

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Application-owned `last_progress_at` is the durable stale-detection signal
- Recovery uses fresh task requeue instead of mutating stale Procrastinate rows
- `pg_try_advisory_lock` guards processing, recovery, and finalization concurrency
- Finalization failures must move imports to an explicit error path instead of leaving them in `PROCESSING`

**Issues Resolved:**

- Imports stuck in `PROCESSING` after worker crashes or deploy restarts are now detected and reclaimed
- Source-exhausted imports can recover their final status transition without replaying rows
- Automated tests now cover stale detection, recovery startup behavior, crash-resume, and duplicate prevention

**Issues Deferred:**

- Periodic reconciliation loop for background recovery cadence
- Configurable recurring recovery interval outside worker startup

**Technical Debt Incurred:**

- None accepted for v1.10
