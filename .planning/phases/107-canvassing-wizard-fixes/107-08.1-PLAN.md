# Plan 107-08.1: Fix Pinning Bug Hiding Auto-Advance / Skip

## Objective

Fix the bug uncovered by plan 107-08 E2E execution: the sort-pinning logic in
`useCanvassingWizard.ts:129-164` re-pins the previously-active household at
the new `currentAddressIndex` after auto-advance or skip, hiding the visible
swap from the user. Internal state advances correctly, toast/vibrate/aria
all fire, but the rendered `HouseholdCard` keeps showing the old address.

This means CANV-01 (auto-advance) and CANV-02 (skip) do not actually deliver
the user-visible behavior the requirements demand, despite passing all unit
tests and the route-mocked E2E spec.

## Root Cause

The `pinnedHouseholdKey` state tracks "what household is the user currently
viewing." It exists so that GPS location updates in distance-sort mode don't
visually shuffle the user's current door out from under them. The pin was
designed for sort-stability, not advance-stability.

When the user records an outcome:
1. `advanceAfterOutcome` → `advanceAddress` → `currentAddressIndex` increments
2. Hook re-renders with the OLD `pinnedHouseholdKey` (the household just acted on)
3. `households` memo finds the pinned household in the ordered list and splices
   it back to the new `currentAddressIndex`
4. `currentHousehold = households[currentAddressIndex]` returns the OLD household
5. User sees the same address even though their action was processed

## Fix Strategy

**Decouple "viewing pin" from "active index pin".**

The pin should release when the user advances through the queue (intentional
navigation), but persist when the underlying ordering changes (GPS update,
sort mode change is already handled).

Concretely: introduce a new effect / mechanism that clears `pinnedHouseholdKey`
when `currentAddressIndex` changes due to advance or skip. The simplest
implementation:

```ts
// Track the previous index so we can detect intentional advances
const prevIndexRef = useRef(currentAddressIndex)
useLayoutEffect(() => {
  if (prevIndexRef.current !== currentAddressIndex) {
    // The active index moved — clear the viewing pin so the new household
    // at currentAddressIndex is what gets displayed.
    setPinnedHouseholdKey(null)
    prevIndexRef.current = currentAddressIndex
  }
}, [currentAddressIndex])
```

Order of effects matters: this `useLayoutEffect` must run BEFORE the
"after-render pin recording" code at lines 166-176, OR the
re-record logic at 170-176 must skip when the index just changed.

**Alternative (cleaner):** make `advanceAddress` and `skipEntry` explicitly
clear the pin themselves rather than relying on an effect. Wrap the calls in
the hook so any advance/skip path releases the pin atomically:

```ts
const advanceAddress = useCallback(() => {
  setPinnedHouseholdKey(null)  // release viewing pin on intentional advance
  storeAdvanceAddress()
}, [storeAdvanceAddress])
```

The wrap-the-action approach is preferred because the intent is explicit
in the action that triggers it, not implicit via index change.
