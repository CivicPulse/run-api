## What Was Built

### Task 1 — `useCanvassingWizard.ts` pin release on intentional navigation

The store's `advanceAddress` and `skipEntry` actions are now destructured under aliases (`storeAdvanceAddress`, `storeSkipEntry`) and re-exposed inside the hook via `useCallback` wrappers that call `setPinnedHouseholdKey(null)` first. The existing `advanceRef` automatically tracks the wrapped version (it depends on the `advanceAddress` identifier in scope), so every call site that goes through `advanceRef.current()` — `maybeAdvanceAfterHouseholdSettled`, `advanceAfterOutcome`, `handleSubmitContact`, `handleSkipAddress` — picks up the new behavior with no additional changes.

The `handleSkipAddress` callback explicitly invokes `skipEntry(id)` (now wrapped) for each entry, so the Skip path also releases the pin synchronously before the index advances.

GPS-update and sort-mode-change pin clearing at lines ~124-127 is unchanged. The post-render "record viewing pin during render" block at ~170-176 is unchanged — it correctly re-pins to whatever is now genuinely at `currentAddressIndex` after the natural-order memo recompute.

### Task 2 — `HouseholdCard.test.tsx` render-path regression-guard

Two tests in a new file:

1. **`rendered address swaps to the next household after a house-level outcome auto-advance`** — uses a 4-entry mock walk-list (3 voters at `house-multi` + 1 at `house-next`), records `not_home` (HOUSE_LEVEL_OUTCOMES) on the active voter, asserts the rendered DOM shows the second household's address.
2. **`rendered address swaps to the next household after Skip`** — same fixture, calls `handleSkipAddress`, asserts the same swap.

The test mounts a tiny `WizardHarness` component that calls `useCanvassingWizard("camp-1", "walk-1")` and renders `wizard.currentHousehold?.address` and `wizard.currentHousehold?.householdKey` into `data-testid` divs. The test asserts on `screen.getByTestId(...).textContent` — the actual DOM text the volunteer would see — not on `result.current.currentAddressIndex`. This deliberately bridges the hook-state vs DOM-render gap that hid the bug from `useCanvassingWizard.test.ts`.

4. Re-applied the fix; re-ran the test; **2/2 passed** again.

The test catches the bug class. It is not a tautology — the failure modes match the exact symptoms documented in `.planning/todos/completed/107-canvassing-pinning-uxgap.md` (DOM stuck on House A even though the hook advanced).

## Decisions Made

See frontmatter `key-decisions`. The two load-bearing ones:

1. **Wrap-the-action over watch-the-effect.** The plan explicitly noted both options (action wrap vs. `useLayoutEffect` on `currentAddressIndex`). Wrap is preferred because (a) intent is explicit at the navigation call site, (b) it avoids an extra render cycle, (c) it doesn't fire on incidental index changes (e.g., the `useLayoutEffect` clamp at line 187 that brings the index back into bounds when the household list shrinks — if pin clearing fired there, distance-mode GPS updates that drop the active household would lose the pin too).
2. **Test in a new file, not the hook test file.** Plan 107-04's executor explicitly removed the `householdKey` assertion from `useCanvassingWizard.test.ts` to "dodge the pinning interaction" (per 107-04 SUMMARY deviation #2). That suggests adding the assertion back to that file would be socially disruptive — and it would still be a hook-level test that doesn't render. The new file at the component-test layer carries the regression-guard at exactly the layer the bug lived: hook → DOM.
