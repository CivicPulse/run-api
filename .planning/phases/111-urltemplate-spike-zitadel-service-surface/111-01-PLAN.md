<objective>
Prove ZITADEL 2.71.x's `urlTemplate` parameter reliably deep-links an authenticated invitee from `auth.civpulse.org` back to `https://run.civpulse.org/invites/<token>` after completing the hosted password-set flow, with the resulting OIDC session visible to `oidc-client-ts` in the SPA.

This is the **gating spike** for Phase 111 and the entire v1.19 milestone. If it fails, Phase 111 exits with `status: blocked (spike failed)`, a `111-SPIKE-VERDICT.md` artifact is produced, and `/gsd-replan-milestone` is proposed to reroute to Option C non-ROPC. No code from plans 02-06 ships until this plan passes.
</objective>

<must_haves>
- [ ] Committed Playwright E2E spike file exists at `web/e2e/invite-urltemplate-spike.spec.ts`
- [ ] Spike runs exclusively through `web/scripts/run-e2e.sh` (logs to `web/e2e-runs.jsonl`)
- [ ] Spike drives full invite -> hosted password-set -> deep-link -> authed backend call flow
- [ ] Pass signal requires ALL THREE: (a) browser lands at `/invites/<token>?zitadelCode=...&userID=...`; (b) `authStore` in `oidc-client-ts` shows a hydrated user; (c) at least one authenticated backend call returns HTTP 200 (e.g. `GET /api/v1/me` or `GET /api/v1/invites/<token>`)
- [ ] Per-run throwaway invitee identity — UUID-suffixed email `spike-<uuid>@civpulse.test`, cleaned up in `afterAll`
- [ ] On failure, Playwright trace (`trace: 'retain-on-failure'`) plus console/network logs are preserved under `web/test-results/`
- [ ] Spike runs against the dev ZITADEL instance only (never prod)
- [ ] urlTemplate shape is LOCKED to: `https://run.civpulse.org/invites/<token>?zitadelCode={{.Code}}&userID={{.UserID}}` (the exact string that ships in EMAIL-03 later)
- [ ] On pass: `111-SPIKE-VERDICT.md` written with `verdict: PASS` and the e2e-runs.jsonl row reference
- [ ] On fail: `111-SPIKE-VERDICT.md` written with `verdict: FAIL`, captured evidence paths, and explicit hand-off to `/gsd-replan-milestone`

<task id="111-01-T2" type="execute">
<action>
Write the `test.beforeAll` setup that provisions a fresh throwaway invitee server-side through the ZITADEL HTTP API.

Steps inside `beforeAll`:
1. Compute `const email = "spike-" + crypto.randomUUID() + "@civpulse.test"` — new every run.
2. Call `getZitadelToken()` to obtain a service-account bearer token.
3. POST to `${process.env.ZITADEL_URL}/v2/users/human` with JSON body:
   ```json
   {
     "username": "<email>",
     "profile": { "givenName": "Spike", "familyName": "Invitee" },
     "email": { "email": "<email>", "isVerified": true }
   }
   ```
   Expect HTTP 201; capture `userId` from the JSON response.
4. POST to `${process.env.ZITADEL_URL}/v2/users/{userId}/invite_code` with JSON body:
   ```json
   { "returnCode": {}, "urlTemplate": "https://run.civpulse.org/invites/<token>?zitadelCode={{.Code}}&userID={{.UserID}}" }
   ```
   Expect HTTP 200; capture `inviteCode` from the JSON response.
5. Store `userId`, `email`, and `inviteCode` on a module-scoped object so later hooks and `afterAll` can reach them.

Use Playwright's built-in `request` APIRequestContext (via `test.beforeAll(async ({ request }) => ...)`) or `fetch`. No new npm dependencies.
</action>

<threat_model>
**STRIDE:**
- **Spoofing (LOW):** The spike creates a real ZITADEL identity. Mitigation: UUID-suffixed email on a `.test` TLD that cannot receive mail; `afterAll` deletes the user; dev-only ZITADEL URL.
- **Tampering (LOW):** Test runs use dev service-account credentials. Mitigation: envs supplied by `run-e2e.sh`; never committed; never touched prod.
- **Repudiation (N/A):** Test artifact, not prod traffic.
- **Information Disclosure (MEDIUM):** Retained traces on failure may contain the generated password. Mitigation: Playwright trace.zip is the only surface; password is random per run and the user is deleted before the trace is reviewed; traces stored under `web/test-results/` which is git-ignored.
- **Denial of Service (LOW):** ZITADEL rate-limit is 50 req/s (PITFALLS §O1); the spike makes ~4 API calls. Mitigation: well below limit; no parallel spike runs.
- **Elevation of Privilege (LOW):** Service-account credentials used for setup must scope down to user-management on the dev org. Mitigation: Plan 05 (SEC-03) audits and narrows the runtime identity; spike can use the same (already-narrowed) identity.

**Blocking severity: none.** No high/critical threats introduced by this plan.
</threat_model>
