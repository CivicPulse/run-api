<objective>
Prove ZITADEL 2.71.x's `urlTemplate` parameter reliably deep-links an invitee from `auth.civpulse.org` back to `https://run.civpulse.org/invites/<token>?zitadelCode=...&userID=...` after completing the hosted password-set flow, carrying the ZITADEL-generated `zitadelCode` + `userID` query params intact.

**Scope narrowed per review B1 (option b).** This spike proves the **landing URL contract only**: the urlTemplate substitution works, the browser arrives at our app at the correct path, the query string carries the two template placeholders populated to concrete values, and a ZITADEL session cookie is set on the `auth.civpulse.org` origin (the precondition for Phase 114's OIDC handoff). Assertions about `authStore` hydration and authed backend calls are OUT OF SCOPE — those require frontend code that processes `zitadelCode` (Phase 114's explicit deliverable: "Any frontend changes to `/invites/<token>` or `/login`"). Moving those assertions here would force Phase 111 to ship Phase 114's code.

This is the **gating spike** for Phase 111 and the entire v1.19 milestone. If it fails, Phase 111 exits with `status: blocked (spike failed)`, a `111-SPIKE-VERDICT.md` artifact is produced, and `/gsd-replan-milestone` is proposed to reroute to Option C non-ROPC. No code from plans 02-06 ships until this plan passes.

**Explicit hand-off to Phase 114:** Phase 114 is responsible for re-verifying the full chain end-to-end (landing URL -> `zitadelCode` exchange -> hydrated `authStore` -> authed API call) once its frontend callback handler ships. Plan 01 produces a PASS verdict here strictly on the landing-URL contract; Phase 114's verification gate re-runs the full flow.
</objective>
