    4. Create `web/src/hooks/useTwilioDevice.ts`:
       - Import { Device, Call } from "@twilio/voice-sdk"
       - Hook signature: `useTwilioDevice(campaignId: string, enabled: boolean)`
       - Internal state: callStatus (TwilioCallStatus), isMuted (boolean), duration (number in seconds), error (string | null)
       - Token fetch: POST to `/api/v1/campaigns/${campaignId}/voice/token` via api client. Store token in ref.
       - Device lifecycle in useEffect (depends on token):
         - `new Device(token, { codecPreferences: [Call.Codec.Opus, Call.Codec.PCMU], closeProtection: "A call is in progress. Leaving will end it." })`
         - Listen for "tokenWillExpire": re-fetch token, call device.updateToken(newToken)
         - Listen for "error": set callStatus to "error", set error message
         - Call device.register()
         - Cleanup: device.destroy()
       - connect(toNumber: string, campaignId: string): async function
         - `const call = await device.connect({ params: { To: toNumber, CampaignId: campaignId } })`
         - Listen for call events: "ringing" -> setCallStatus("ringing"), "accept" -> setCallStatus("open") + start interval timer, "disconnect" -> setCallStatus("closed") + stop timer, "error" -> setCallStatus("error")
         - "mute" event: setIsMuted(call.isMuted())
       - disconnect(): callRef.current?.disconnect()
       - toggleMute(): callRef.current?.mute(!isMuted)
       - Duration timer: useRef for interval ID. Start on "accept", clear on "disconnect"/"error". Increment every 1000ms.
       - Return: { callStatus, isMuted, duration, error, connect, disconnect, toggleMute, isReady: boolean (device registered) }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Token API | Frontend requests Access Token; token scoped to user identity |
| Browser -> Twilio SRTP | WebRTC audio encrypted end-to-end by Twilio SDK |
| User -> Microphone | Browser permission gate for audio access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-10 | Information Disclosure | Access Token in browser memory | accept | Token TTL 600s, contains no secrets, identity-bound; standard Twilio pattern |
| T-91-11 | Tampering | Client-side DNC check bypass | mitigate | DNC is also checked server-side in TwiML handler (Plan 02); client check is UX convenience only |
| T-91-12 | Tampering | Client-side calling hours bypass | mitigate | Calling hours also checked server-side in TwiML handler (Plan 02); client check is UX convenience only |
| T-91-13 | Denial of Service | Microphone permission denial loop | accept | Graceful fallback to tel: mode with toast; user controls browser permissions |
| T-91-14 | Spoofing | Token identity mismatch | mitigate | Token identity derived from authenticated user_id on server; client cannot change identity |
</threat_model>
