  <action>
Create `scripts/seed.py` that populates a complete demo dataset. The script runs via `docker compose exec api python scripts/seed.py`.

**Idempotency approach:** At the start, query for a Campaign with name "Macon-Bibb Demo Campaign". If it exists, print "Seed data already exists, skipping." and exit 0. If not, proceed with full creation.

**Important:** The script connects directly using SQLAlchemy async. Since it runs inside the API container, it uses the same DATABASE_URL from the environment. Create a standalone async engine (do NOT import app.db.session — that module creates a global engine at import time which is fine for the app but the seed script should control its own connection). Use `create_async_engine(os.environ["DATABASE_URL"])` and `async_sessionmaker`.

**Data to create (all with explicit UUIDs via uuid4() for FK consistency):**

1. **Users** (3): Fabricated users with ZITADEL-style IDs (random strings). These represent the demo campaign owner, a manager, and a volunteer. No actual ZITADEL accounts needed — seed data is for API/DB testing only.

2. **Campaign** (1): "Macon-Bibb Demo Campaign", type="general", status="active". Set created_by to the owner user's ID.

3. **CampaignMembers** (3): Owner (role="owner"), manager (role="manager"), volunteer (role="volunteer").

4. **Voters** (~50): Fabricated names with realistic Macon-Bibb County GA coordinates.
   - Spread across neighborhoods: Ingleside (32.845, -83.635), Vineville (32.843, -83.648), Pleasant Hill (32.835, -83.620), downtown (32.840, -83.632), Bellevue (32.830, -83.645)
   - Mix of party affiliations (D, R, NP, L), voter statuses (active, inactive)
   - Include lat, lng fields. Also set the `geom` column using `func.ST_SetSRID(func.ST_MakePoint(lng, lat), 4326)` from geoalchemy2/sqlalchemy.
   - Each voter gets a VoterPhone and VoterEmail contact record.

5. **Turfs** (3): Named after real Macon-Bibb neighborhoods.
   - Create WKT polygon boundaries covering those neighborhoods using `func.ST_GeomFromText(wkt, 4326)`.
   - Approximate rectangular polygons that encompass the voter coordinates in each area.

6. **Walk Lists** (2): One active, one completed. Each with ~10-15 WalkListEntries pointing to voters in the corresponding turf.

7. **Survey Script** (1): "Voter Outreach Survey" with 3-4 questions (multiple_choice and yes_no types). Include some SurveyResponses tied to voters.

8. **Volunteers** (5): Linked to the campaign with various skills.

9. **Shifts** (2): One past (completed), one upcoming. With ShiftVolunteer records for the past shift including check_in/check_out times.

10. **Phone Bank Session** (1): With a CallList, some CallListEntries, and a few completed call results recorded as VoterInteractions.

11. **VoterInteractions**: Create ~15-20 interactions across types:
    - DOOR_KNOCK results (supporter, not_home, refused) for walk list activity
    - PHONE_CALL results (answered, no_answer, voicemail) for phone bank activity
    - SURVEY_RESPONSE for survey answers
    This ensures dashboard endpoints have data to aggregate.

**Structure:** Use a single `async def main()` function that creates all data in order (respecting FK dependencies). Use `session.add_all()` for batch inserts, `await session.flush()` after each entity group to get generated IDs. Wrap everything in a transaction.

**Run at the bottom:**
```python
if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

**Import models directly** from `app.models.*` — they're available since the script runs inside the container with the app code mounted.

Use `from sqlalchemy import func, text` for PostGIS functions. Use `from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession`.

Ensure all timestamps (created_at, updated_at) are set explicitly using `datetime.now(UTC)` to avoid NULL issues when server_default isn't triggered.
