    Order the triage by impact: work the historical flake hit list first (these specs had 12-59 historical failing runs — extra budget), then the remaining baseline rows.

    **A. Per-spec procedure (D-04 3x rerun rule):**

    For each failing spec listed in baseline:

    1. Run the spec once through the wrapper:
       ```bash
       cd web && ./scripts/run-e2e.sh <spec-filename>
       cd ..
       ```

    2. If it PASSES on this first retry (was a transient flake) → run 3x total to confirm:
       ```bash
       cd web
       for i in 1 2 3; do
         echo "=== Run $i/3 ==="
         ./scripts/run-e2e.sh <spec-filename> || echo "FAILED on run $i"
       done
       cd ..
       ```
       - 3/3 pass → still investigate root cause briefly (race, selector, state leak). Either identify and fix the root cause OR, if no cause is findable within the time box, DELETE per D-04 (flake = broken).
       - 1-2 passes → confirmed flaky, fix root cause (common: auth state leak, shared fixture race, `waitFor` replaced with proper `expect().toBeVisible()`) OR delete.
       - 0/3 passes → hard fail, apply the standard 15-min fix box.

    3. If it FAILS on the first retry → standard 15-min fix box:
       - Pitfall 5 check: if spec matches `canvassing*`, `walk-list*`, `field-mode.volunteer.*`, `*map*`, `leaflet*`, `offline*`, `sync*` — DELETE with `PHASE-106-DELETE:` body line referencing phase 107/108/109/110.
       - Otherwise attempt a mechanical fix: stale selectors (shadcn upgrades), timing (`waitFor` → `expect(...).toBeVisible()`), auth state (clear and re-run), fixture mismatches.
       - If the test fails due to a real product bug (D-08): DELETE with justification, file a todo under `.planning/todos/pending/e2e-<slug>.md`.
       - Time box exceeded → DELETE with justification.

    4. Pitfall 2 sanity check (only if a spec passes solo but fails in full suite): re-run at lower worker count to confirm concurrency flake:
       ```bash
       cd web && ./scripts/run-e2e.sh --workers 4 <spec-filename>
       cd ..
       ```
       Concurrency flakes that only reproduce at `--workers 16` are candidates for fix-the-test (reduce shared state) or DELETE. Do NOT change the default worker count in `playwright.config.ts` (D-09 infra rewrite fence).

    5. Pitfall 6 sanity check: if the wrapper aborts with "Strict policy verification failed", run the remediation it prints and re-attempt. Do not bypass.
