<objective>
Close MAP-02: when a volunteer opens the household list (the
`DoorListView` bottom sheet triggered by the "All Doors" button in
`canvassing.tsx`), the list must be fully visible and interactable. The
map underneath must not overlay the list via stacking context, must
not intercept taps via pointer-events, and must not rebuild its Leaflet
instance when the sheet toggles (so the phase-108 panTo state survives).

Root cause confirmed by code inspection:
- `DoorListView` uses Radix Sheet (`side="bottom"`) which renders a
  fixed-position overlay. Radix overlays default to `z-index: 50`.
- Leaflet's default pane z-index stack runs 200 (tile) → 700 (popup).
  The `.leaflet-container` also creates its own stacking context via
  its positioning, so on iOS Safari especially the Sheet can render
  BEHIND Leaflet tooltip/popup panes even though the Sheet has z-index 50.
- Even when stacking is correct, the map container beneath the sheet
  can still capture touch events that bleed through the sheet edges on
  mobile viewports because the sheet is `max-h-[80dvh]` (not full
  screen).

Fix: (a) add a CSS rule that guarantees Radix Sheet overlay + content
z-index stays above Leaflet's highest pane; (b) while `listViewOpen`
is true, wrap the map section in a wrapper div that gets an `--inert`
modifier class disabling pointer-events on all descendants (but NOT
`display: none`, so the Leaflet instance stays mounted and `panTo`
state from 108-03 survives).

Purpose: reported volunteer bug — in the field, tapping household
cards in the bottom sheet sometimes hits the map instead of the card,
especially at the top edge of the sheet where it overlaps the map's
visible area.

Output: CSS rule + wrapper div + inert class toggled by `listViewOpen`.
</objective>
