<objective>
The phase 110 audit (Plan 110-01) established that the current offline queue
has a stable client-side `QueueItem.id` (crypto.randomUUID) but **never sends
it to the server**. So if a queued door_knock partially replays (network drops
after the server commits but before the client sees 201), the retry creates a
duplicate. OFFLINE-01 demands exactly-once delivery — this plan delivers it.

**Output:** Client payload, server schema, DB migration, and integration test
that together make duplicate replays a no-op (return 409, client removes from
queue — the existing isConflict path already handles this).
</objective>
