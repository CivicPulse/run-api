  <name>Task 1: Implement SurveyService and survey API endpoints</name>
  <files>app/services/survey.py, app/api/v1/surveys.py, app/api/v1/router.py</files>
  <action>
1. Create `app/services/survey.py` with SurveyService:
   - Composition: `__init__` creates `self._interaction_service = VoterInteractionService()`

   Script CRUD:
   - `create_script(session, campaign_id, data: ScriptCreate, user_id) -> SurveyScript`: Create with status=DRAFT
   - `get_script(session, script_id) -> SurveyScript | None`
   - `list_scripts(session, campaign_id, status_filter=None, cursor=None, limit=20) -> tuple[list, str|None, bool]`: Paginated, optional status filter
   - `update_script(session, script_id, data: ScriptUpdate) -> SurveyScript`:
     - If updating status: enforce transitions only draft->active, active->archived. No backward transitions. Raise ValueError on invalid transition.
     - If updating title/description: only allowed when status=DRAFT. Raise ValueError if trying to edit active/archived script metadata.
   - `delete_script(session, script_id) -> None`: Only draft scripts can be deleted. Active/archived scripts have responses referencing them.

   Question CRUD (lifecycle-gated):
   - `add_question(session, script_id, data: QuestionCreate) -> SurveyQuestion`:
     - CRITICAL: Verify script.status == DRAFT. Raise ValueError if not.
     - Validate options based on question_type:
       - multiple_choice: require `options.choices` as list of 2-10 non-empty strings
       - scale: require `options.min` < `options.max`, both integers, max range 1-100
       - free_text: options should be None (ignore if provided)
     - If position not specified, append after last question (max position + 1)
   - `update_question(session, question_id, data: QuestionUpdate) -> SurveyQuestion`: Only if parent script is DRAFT. Validate options if type changed.
   - `delete_question(session, question_id) -> None`: Only if parent script is DRAFT
   - `reorder_questions(session, script_id, question_ids: list[uuid.UUID]) -> list[SurveyQuestion]`: Set position based on order in list. Only if DRAFT.
   - `list_questions(session, script_id) -> list[SurveyQuestion]`: Ordered by position

   Response recording:
   - `record_response(session, campaign_id, script_id, data: ResponseCreate, user_id) -> SurveyResponse`:
     - Verify script exists and is ACTIVE (not draft or archived)
     - Verify question belongs to the script
     - Validate answer_value against question type:
       - multiple_choice: answer must be one of the choices
       - scale: answer must be integer within min/max range
       - free_text: any non-empty string
     - Create SurveyResponse row
     - Return the response
   - `record_responses_batch(session, campaign_id, script_id, voter_id, responses: list[ResponseCreate], user_id) -> list[SurveyResponse]`:
     - Record multiple responses (one per question for a voter completing a script)
     - After recording all responses, emit a single SURVEY_RESPONSE interaction event:
       `interaction_type=SURVEY_RESPONSE, payload={"script_id": str(script_id), "question_count": len(responses)}`
     - This is the dual storage pattern: queryable responses in survey_responses + audit trail in voter_interactions
   - `get_voter_responses(session, campaign_id, voter_id, script_id) -> list[SurveyResponse]`: Get all responses for a voter on a script, ordered by question position

2. Create `app/api/v1/surveys.py` with router:
   Script endpoints:
   - POST `/campaigns/{campaign_id}/surveys` -- Create script (require_role("manager"))
   - GET `/campaigns/{campaign_id}/surveys` -- List scripts, optional status filter (require_role("volunteer"))
   - GET `/campaigns/{campaign_id}/surveys/{script_id}` -- Get script with questions (require_role("volunteer"))
   - PATCH `/campaigns/{campaign_id}/surveys/{script_id}` -- Update script (require_role("manager"))
   - DELETE `/campaigns/{campaign_id}/surveys/{script_id}` -- Delete draft script (require_role("manager"))

   Question endpoints:
   - POST `/campaigns/{campaign_id}/surveys/{script_id}/questions` -- Add question (require_role("manager"))
   - PATCH `/campaigns/{campaign_id}/surveys/{script_id}/questions/{question_id}` -- Update question (require_role("manager"))
   - DELETE `/campaigns/{campaign_id}/surveys/{script_id}/questions/{question_id}` -- Delete question (require_role("manager"))
   - PUT `/campaigns/{campaign_id}/surveys/{script_id}/questions/order` -- Reorder questions (require_role("manager"))

   Response endpoints:
   - POST `/campaigns/{campaign_id}/surveys/{script_id}/responses` -- Record batch responses for a voter (require_role("volunteer")). Request body: {voter_id, responses: [{question_id, answer_value}]}
   - GET `/campaigns/{campaign_id}/surveys/{script_id}/voters/{voter_id}/responses` -- Get voter's responses for a script (require_role("volunteer"))

3. Update `app/api/v1/router.py`:
   - Add import for surveys
   - `router.include_router(surveys.router, tags=["surveys"])`
   - NOTE: Plan 02 also modifies router.py. If running in parallel, the executor should merge both changes. If running after Plan 02, just add the surveys import and include_router line.

Follow existing patterns: Depends(require_role(...)), RFC 9457 errors, cursor-based pagination.

Create `tests/integration/test_canvassing_rls.py` following the exact pattern from existing RLS integration tests (likely in tests/integration/test_rls.py or similar from Phase 1-2).

The pattern:
- Use the existing test database fixtures (superuser session and app_user session from conftest.py)
- Superuser creates data in Campaign A and Campaign B
- App user sets RLS context to Campaign A via `SET LOCAL app.current_campaign_id = '{campaign_a_id}'`
- App user queries each table and verifies only Campaign A data is visible

Test cases:

1. `test_turf_rls_isolation`:
   - Superuser inserts a turf in Campaign A and a turf in Campaign B
   - App user with Campaign A context: SELECT from turfs returns only Campaign A's turf
   - App user with Campaign B context: returns only Campaign B's turf

2. `test_walk_list_rls_isolation`:
   - Superuser creates walk lists in both campaigns
   - Verify isolation on walk_lists table

3. `test_walk_list_entries_rls_isolation`:
   - Superuser creates walk list entries (via walk lists in different campaigns)
   - Verify entries only visible through their parent walk list's campaign
   - This tests the subquery RLS policy: `walk_list_id IN (SELECT id FROM walk_lists WHERE campaign_id = ...)`

4. `test_walk_list_canvassers_rls_isolation`:
   - Similar subquery-based RLS test for canvasser assignments

5. `test_survey_script_rls_isolation`:
   - Superuser creates scripts in both campaigns
   - Verify isolation

6. `test_survey_questions_rls_isolation`:
   - Tests subquery RLS through survey_scripts parent
   - Questions only visible when parent script's campaign matches context

7. `test_survey_responses_rls_isolation`:
   - Superuser creates responses in both campaigns
   - Verify direct campaign_id RLS isolation

8. `test_voter_geom_column_exists`:
   - Verify the geom column was added to voters table
   - Insert a voter with lat/long, verify geom is populated (or can be set)

Read the existing RLS test file first to match the exact fixture names, session setup, and assertion patterns used in this project. The test should use `pytest.mark.asyncio` and async test functions.

Each test should:
- Use `text()` for raw SQL (RLS operates at SQL level, not ORM level for these tests)
- Clean up after itself or use transaction rollback
- Assert both positive (correct campaign visible) and negative (other campaign not visible) cases
