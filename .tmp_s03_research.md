# M002-62wd19/S03 — Research

**Date:** 2026-04-03

## Summary

S03 owns the canvassing-side requirements in the milestone context: R026/R027/R028 are the direct targets, and the route already contains most of the structural seams needed to deliver them. The current field canvassing flow in `web/src/routes/field/$campaignId/canvassing.tsx` can already load the active household, record per-voter outcomes through `useCanvassingWizard`, open the inline survey sheet for contact outcomes, bulk-mark the rest of a household as `not_home`, and skip the current address into a revisit queue. The important finding is that the route currently treats survey capture and door-knock persistence as two separate save boundaries: `handleOutcome()` posts the door-knock immediately, then `InlineSurvey` saves survey responses independently via `useRecordResponses`, and canvassing advances afterward. Unlike S01 phone banking, there is no single authoritative submit path that bundles outcome + notes + survey answers for one doorstep interaction.

The backend currently only supports `DoorKnockCreate { voter_id, walk_list_entry_id, result_code, notes? }` and records a `door_knock` interaction plus entry status updates in `app/services/canvass.py`. There is no canvassing-specific contract for `survey_responses`, no household-level helper for “one meaningful contact, rest of address visited,” and no explicit verification around skip-to-queue semantics beyond the broad volunteer spec. The natural implementation path is therefore: extend the canvassing save contract first, then refactor the route/wizard to use one authoritative submission boundary for contact outcomes, and finally tighten household flow + skip behavior with focused tests and deterministic FIELD fixtures.

## Recommendation

Follow the S01 pattern: make canvassing contact recording authoritative at one submit boundary instead of persisting outcome first and survey later. Concretely, extend the door-knock API/service contract to accept optional notes plus ordered survey responses (and likely a survey-complete boolean), have the backend write the `door_knock` interaction and survey responses in one flow, then refactor `web/src/routes/field/$campaignId/canvassing.tsx` so contact outcomes open a controlled `InlineSurvey` sheet that submits one final payload only when the volunteer finishes notes + survey.

Keep the S02 ordered-household seam intact. `useCanvassingWizard` should remain the owner of household progression, pinned current door semantics, bulk-not-home handling, and skip-to-queue behavior. S03 should add address-flow rules on top of that seam rather than introducing a second navigation model. The planner should build/prove the backend payload first because it unblocks the frontend from collecting complete canvassing records and reduces the risk of repeating the pre-S01 split-save failure mode.

## Implementation Landscape

### Key Files

- `web/src/routes/field/$campaignId/canvassing.tsx` — Main volunteer canvassing route. Current seam for outcome selection, survey opening, post-survey advancement, bulk not-home prompt, and skip button. This is where the split-save flow currently exists and where the final authoritative submit UX should land.
- `web/src/hooks/useCanvassingWizard.ts` — Owns ordered households, active entry selection, optimistic outcome recording, auto-advance, bulk-not-home, and `handleSkipAddress()`. Best place for progression rules and for adding a dedicated “submit contact record” mutation path if the route should stay thinner.
- `web/src/hooks/useCanvassing.ts` — Current data/mutation layer for enriched entries, door-knock POSTs, and entry-skip PATCH. This will need the door-knock mutation contract extended if canvassing is to submit notes + survey responses together.
- `app/api/v1/walk_lists.py` — Declares the canvassing endpoints: enriched entries, skip-entry PATCH, and door-knock POST. The authoritative canvassing save contract change starts here.
- `app/services/canvass.py` — Current door-knock service. Today it records a `door_knock` interaction, marks the entry visited, increments `visited_entries`, and returns `DoorKnockResponse`. It is the backend seam for bundling survey response persistence with the door-knock transaction.
- `app/schemas/canvass.py` — Defines `DoorKnockCreate` / `DoorKnockResponse` and enriched-entry response shape. It currently supports `notes` only; planner should expect this schema to grow for S03.
- `web/src/components/field/InlineSurvey.tsx` — Reusable survey sheet already supports both `mode="direct-save"` and `mode="controlled"`. S03 can reuse the controlled mode already introduced for S01 phone banking rather than inventing a new survey modal.
- `web/src/routes/field/$campaignId/phone-banking.tsx` — Canonical reference for the S01 authoritative submit pattern. It uses `InlineSurvey mode="controlled"` and waits to call `submitCall()` until notes + survey answers are complete.
- `web/src/components/field/HouseholdCard.tsx` — Renders the current address, resident cards, and skip button. If S03 needs clearer address-level affordances (“rest of household visited”, “come back later”, etc.), this is the presentation seam.
- `web/src/components/field/VoterCard.tsx` — Per-resident outcome grid and completed/skipped state. Useful if the planner decides to surface explicit contact-vs-noncontact status per resident after one household interaction.
- `web/src/components/field/DoorListView.tsx` — All Doors sheet. It already reflects `Pending/Visited/Skipped` from store state and is the main UI evidence for skip-back-into-queue behavior.
- `web/e2e/helpers.ts` — Existing deterministic canvassing fixture builders. `createDisposableCanvassingSurveyFixture()` already creates a walk list with an active survey script; this is the best fixture seam for FIELD-07/S03-style recording assertions.
- `web/e2e/field-mode.volunteer.spec.ts` — Existing volunteer canvassing coverage. FIELD-04 only records simple outcomes; FIELD-07 proves inline survey opens and can submit, but does not verify notes persistence, realistic rest-of-household handling, or skip-back-into-queue semantics.
- `tests/unit/test_canvassing.py` — Current backend service tests for door-knock persistence and visited counts. This is the right place to prove new payload fields and combined save behavior.
- `web/src/hooks/useCanvassingWizard.test.ts` — Existing focused hook test seam. Good place to add address-flow regression tests without requiring full route rendering.

### Build Order

1. **Prove the authoritative canvassing save contract first.** Extend `DoorKnockCreate` / `record_door_knock()` to support the complete interaction payload needed by S03. This is the highest-risk seam because everything else depends on whether outcome + notes + survey answers can be saved together.
2. **Refactor the canvassing route to use controlled survey submission.** Mirror the S01 phone-banking pattern in `web/src/routes/field/$campaignId/canvassing.tsx` using `InlineSurvey mode="controlled"`, so contact outcomes do not mutate the backend until the volunteer finishes the interaction.
3. **Tighten household progression semantics.** Adjust `useCanvassingWizard` and/or route-level handlers so one meaningful contact can complete the active resident with notes/survey while remaining residents can be handled appropriately (bulk not-home, visited-without-contact-detail, or similar explicit logic chosen during planning) without trapping the volunteer.
4. **Lock down skip-to-queue behavior and door-list visibility.** Verify `handleSkipAddress()` plus `DoorListView` preserve skipped addresses as revisitable queue items and do not silently abandon them.
5. **Add deterministic verification.** Expand unit tests and Playwright FIELD coverage only after the contract/flow is stable so the final proof matches the real implementation boundary.

## Constraints

- S02 established the ordered-household array in `useCanvassingWizard` as the authoritative route contract. S03 should keep map/current card/All Doors synchronized through that same array rather than inventing separate queue state.
- `InlineSurvey` already has two modes. Reusing `mode="controlled"` follows the existing pattern and avoids hand-rolling another canvassing-specific notes/survey sheet.
- The current backend door-knock schema only supports `notes`; survey persistence today is a separate surveys API call. Any “single authoritative submit” design for canvassing requires backend contract work, not just frontend wiring.
- Project instruction: use `uv` for Python commands; do not rely on bare `python` during execution or verification.

## Common Pitfalls

- **Repeating the split-save failure from pre-S01 phone banking** — if canvassing still records the outcome before survey/notes submission, volunteers can produce partial doorstep records and progression bugs will remain subtle. Keep one final submit boundary.
- **Breaking S02 pinned-door semantics** — distance ordering now pins the active household. Any requeue/skip/completion logic must preserve `currentAddressIndex` assumptions used by the map and All Doors sheet.
- **Treating skipped entries as completed** — `DoorListView` marks a household `Visited` only when all entries are completed, and `Skipped` only when all entries are skipped. Mixed household states are currently collapsed to `Pending`; if UX depends on more nuance, tests must assert the intended intermediate behavior.
- **Assuming offline behavior is the proof path** — `useCanvassingWizard` still queues `door_knock` on `TypeError`, but M002 is explicitly live-connected. S03 verification should focus on truthful online behavior first.

## Open Risks

- The exact backend representation for “one person contacted, rest of address appropriately handled” is still ambiguous. Current primitives are per-entry `visited`/`skipped` plus outcome interactions; the chosen S03 rule may need a deliberate compromise unless product semantics are clarified during planning.
- `field-mode.volunteer.spec.ts` currently proves only broad canvassing success. Additional deterministic fixture work may be required if the planner wants explicit multi-resident household assertions instead of relying on seeded data.
